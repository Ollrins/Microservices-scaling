
# Домашнее задание к занятию «Микросервисы: масштабирование»

# Задача 1: Кластеризация

## Предлагаемое решение

**Docker Swarm** + **Docker Secrets** + **Prometheus + AlertManager** (для автомасштабирования)

---

## Соответствие требованиям (подробно)

| Требование | Как реализуется в Docker Swarm |
|------------|--------------------------------|
| **Поддержка контейнеров** | Swarm нативно работает с Docker-контейнерами. Управление через стандартный CLI: `docker service create`, `docker stack deploy` |
| **Обнаружение сервисов и маршрутизация** | Встроенный **Routing Mesh** балансирует трафик между всеми репликами сервиса на всех нодах. Внутренний DNS позволяет сервисам обращаться друг к другу по имени (например, `backend:8080`) без привязки к IP |
| **Горизонтальное масштабирование** | Реализуется командой `docker service scale service_name=N` или через изменение секции `replicas` в compose-файле. Масштабирование возможно как вручную, так и программно через API |
| **Автоматическое масштабирование** | В Swarm нет встроенного HPA, но реализуется через связку **Prometheus + AlertManager**: сбор метрик с контейнеров → превышение порога → вебхук в Docker API → изменение количества реплик |
| **Разделение внешних и внутренних ресурсов** | Создаются **overlay-сети** для внутренней коммуникации (БД, брокеры, кеши). Доступ извне открывается только через опубликованные порты (например, `-p 80:80` для API-шлюза) |
| **Конфигурация через переменные среды** | Обычные настройки передаются через `environment` в compose-файле или через `configs` |
| **Безопасное хранение чувствительных данных** | Используются **Docker Secrets**: пароли, ключи и токены хранятся в зашифрованном виде, монтируются в контейнеры как временные файлы (например, `/run/secrets/db_password`), не отображаются в логах и не передаются через переменные окружения, где их можно перехватить |
---

## Обоснование выбора Docker Swarm

### 1. Достаточность функционала
Docker Swarm закрывает **все требования** задачи без избыточной сложности:
- Управление контейнерами "из коробки"
- Встроенный механизм обнаружения сервисов и маршрутизации
- Поддержка горизонтального масштабирования
- Встроенные механизмы безопасности и хранения секретов

### 2. Простота внедрения и поддержки
- Является частью Docker Engine, не требует установки дополнительного ПО
- Использует стандартные Dockerfile и compose.yml
- Не требует выделенной команды администраторов, как Kubernetes

### 3. Надёжность для production-нагрузок
- Поддерживает режим HA (High Availability) с несколькими менеджерами
- Автоматически перезапускает упавшие контейнеры
- Распределяет нагрузку между нодами через Routing Mesh

### 4. Гибкость и перспектива развития
- В любой момент можно мигрировать на Kubernetes без переписывания Dockerfile
- Интегрируется со стандартным CI/CD (GitLab CI, Jenkins, GitHub Actions)

# Задача 2: Распределённый кеш (Redis Cluster)

## Постановка задачи

Построить Redis Cluster из **3 шард**, каждая с **3 репликами** (всего 9 узлов). Размещение — на 3 виртуальных машинах.

---

## Предлагаемое решение

**Redis Cluster** с распределением узлов по трём VM для обеспечения отказоустойчивости и автоматического восстановления при сбоях.

---

## Архитектура и распределение

Redis использует 16384 хэш-слотов, которые распределяются между шардами:

| Шарда | Слоты | Мастер | Реплики |
|-------|-------|--------|---------|
| Shard 1 | 0 – 5460 | VM 1 | VM 2, VM 3 |
| Shard 2 | 5461 – 10922 | VM 2 | VM 1, VM 3 |
| Shard 3 | 10923 – 16383 | VM 3 | VM 1, VM 2 |

### Схема размещения на VM

| VM | Узлы |
|----|------|
| **VM 1** | Мастер Shard 1 + Реплика Shard 2 + Реплика Shard 3 |
| **VM 2** | Мастер Shard 2 + Реплика Shard 1 + Реплика Shard 3 |
| **VM 3** | Мастер Shard 3 + Реплика Shard 1 + Реплика Shard 2 |

---

## Обоснование архитектуры

| Принцип | Почему это важно |
|---------|------------------|
| Мастер каждой шарды на отдельной VM | Падение одной VM не уничтожает все мастер-ноды |
| Реплики шарды распределены по другим VM | При падении мастера реплика на другой VM автоматически становится новым мастером |
| Ни одна VM не содержит все реплики шарды | Исключается потеря всех копий данных при отказе одной VM |

---

## Отказоустойчивость

| Сценарий | Как кластер реагирует |
|----------|----------------------|
| Падение 1 VM | Реплики потерянного мастера переизбирают нового мастера. Кластер продолжает работу |
| Падение 2 VM | У каждой шарды остаётся минимум одна реплика на выжившей VM. Кластер перестраивается |
| Падение всех 3 VM | Полная недоступность. Восстановление из RDB/AOF-бекапов |

**Механизм восстановления:** Redis Cluster использует Gossip-протокол для обнаружения сбоев. При недоступности мастера реплики инициируют голосование и выбирают нового мастера за 5–15 секунд (настраивается параметром `cluster-node-timeout`).

---

## Требования к инфраструктуре

| Требование | Описание |
|------------|----------|
| **Сеть** | Все 9 узлов должны видеть друг друга. Нужны два порта на узел: клиентский (6379) и Cluster Bus (6379+10000) для служебного обмена |
| **Хранилище** | У каждого узла должен быть отдельный постоянный том (Persistent Volume) для RDB/AOF-файлов, чтобы данные сохранялись при перезапуске |
| **Память** | Настроить политику вытеснения (`maxmemory-policy allkeys-lru`), чтобы кеш не блокировал запись при заполнении |
| **Мониторинг** | Отслеживать состояние кластера, заполнение памяти, отставание реплик через Prometheus Redis Exporter |

---

## Развёртывание кластера

### Шаг 1. Запуск узлов

На каждой VM запускается по 3 экземпляра Redis (на портах 7000, 7001, 7002) с включённым режимом кластера.

### Шаг 2. Объединение в кластер

После запуска всех 9 узлов выполняется команда:

```bash
redis-cli --cluster create \
  VM1:7000 VM2:7000 VM3:7000 \
  VM1:7001 VM2:7001 VM3:7001 \
  VM1:7002 VM2:7002 VM3:7002 \
  --cluster-replicas 2
```
